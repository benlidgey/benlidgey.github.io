Burning the berms

Last Sunday I went mountain biking with my son. I’ve not been mountain biking properly for ages, but loved it. The kit has changed so much! My old mountain bike was a fairly standard shape frame, with similar gear ratios to hybrids, and with slightly wider tyres. The Whyte I hired from Afan Valley Bike Shed was a lovely bit of kit. Wider tyres, more upright and sat back frame geometry, only 1 front ring and the lowest gear at the back looked larger than the large chain ring on my road bike. It made cycling up the side of a mountain easy — so much so we did it twice. We enjoyed the Widowmaker so much the first time, that the extra effort and leg burns was worth doing to have a second go.

It was a great day out, lovely scenery and sunshine, and had a great time with my son. Next time I might have to hire a full sus.

Repos for microservices

Had a debate this week about how to manage repositories for micro-services. There are several options with pros and cons. This article “Code repository for micro-services: mono repository or multiple repositories” outlines many of the pros and cons nicely.

My preference is to have a repo for each microservice. The service can be built, released and deployed in a controlled manner. One discussion was that libraries that might be needed to connect with the service might be in a separate repo — the solution I think is to have both the client and server code in the repo. That way the version of each can be managed.

Releases

We had another discussion this week about release planning, software releases, builds, deployment and releasing code to servers. There is often an overloading of words like “release”:

  • A release — a set of features that will be discussed and delivered to the customer (“the June release will contain the amazing new feature”)
  • A build of a component or service (“can you release the change you have just committed?”)
  • Putting a build of a component or service onto a server (“can you release version 1.2 to the test server?”)

This has been complicated by the inconsistent naming in Jira — the Agile boards allow the definition of “releases”, with stories being dragged into a release. These are called “fix version”. You can get release notes for the tickets in a release/fix version. Should these “release notes” show the stories and bugs in a version of the software, or for the functionality to be discussed with the customer. The Jira Agile boards don’t help as the release/fix version is overwritten if you drag and drop a ticket, although you can have more than one set on a ticket.

A few weeks ago in S04E04 Positive introspection and adaption I thought about the overloaded use of build too.

As we started a new repo with CI/CD that can automatically update Jira tickets (if the Jira ref is in the commit), we wanted to track the tickets closed in a set of commits. At the same time we need to manage the tickets planned for a business release that can be monitored and progress communicated.

We decided how it was all going to work, and time will tell if that is the right way to go. Probably more on that at a later date.

Cycling to city centre offices

We are looking at a potential office move to the city centre. One thing that struck me is the number of adverts that mention car parking but very few mention (secure) cycle parking. It sounds like the council need to put a lot more effort into sustainable transport in the city.
