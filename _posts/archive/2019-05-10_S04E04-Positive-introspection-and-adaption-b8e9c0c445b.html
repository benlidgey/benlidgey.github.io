Introspection and imposters

I read this article “Success is a driver of imposter syndrome” this week and it reminded me of a time in a previous job where my ability and skills were questioned by other to the extent that I ended up questioning my own abilities. I went through a long period of introspection and (thankfully) came out realising my own strengths and skills. This ultimately made me stronger and more confident. It also made me more aware of the default behaviours I need to be aware of, and how to change my behaviour in certain situations. For example, one of my traits is that I have a dry, sarcastic sense of humour which may not be appropriate in certain circumstances. So, when I feel I am about to say something I regard as funny I double check if it the others I am with would feel the same, or if it may be mis-construed.

I realise it is important to celebrate successes (no matter how trivial it may seem) with the team, but to also recognise and celebrate my own successes. This week I have achieved some hard rides on the turbo trainer, completed some definite DIY tasks and had good conversations with colleagues on how to improve the way we work, and have learned things as part of it.

Daily scrums and status meetings

Yesterday one of the team challenged the purpose of one of the daily scrums. The conversations often go on longer than perhaps they should, and are not always relevant to all of the team. We then had a good debate about the point of a daily scrum, and the difference between what it aims to achieve and a “status meeting”. The Scrum Guide describes it as:

The Development Team uses the Daily Scrum to inspect progress toward the Sprint Goal and to inspect how progress is trending toward completing the work in the Sprint Backlog.

We tended to treat it as a status meeting, with people going through what they are working on, rather than focussing on what needs to be done to meet the sprint goal. Scrum Myths: Daily Scrum Is Not a Status Meeting is an interesting article on the differences, and again highlights the focus on transparency, collaboration and outcomes.

There are other reasons that I think contribute to the length of the daily scrum, including the large team size and that there are several different streams of work happening in the same team/sprint. This suggests the need to split the work and team into the different streams and different daily scrums. The work is probably not that simple to split, but it is a good discussion to have in the next retro and to experiment.

Debates without empirical evidence

Another debate this week is the relative priority of delivering user stories and delivering technical improvements. This kicks off the debate about the impact on delivering user stories if people work on technical improvements — technical improvement work may delay the completion of user stories, while technical improvements may improve the quality and speed at which user stories can be delivered. Without decent measurements on velocity, metrics like test coverage and rework it still remains a bit of guesswork.

To build or deploy, that is the question

The final debate this week has been on the build process, and the overloaded use of “build” to mean the building of some code and the deployment of some code to an environment (where sometimes a new build needs to be made for a new environment).

A build is when a known state of code is built using a known set of tools. This typically means the code is from a known bracnch and tagged. The code is then built once using Jenkins and the built artifacts stored somewhere like Artifactory or Docker Registry. Builds include information on the release version, build number, changes (commits or Jira) included in the build.

A deployment is then when a known build is deployed onto an environment with config for that environment. The config should also be stored and managed in source control — that way we can tag the config for that deployment allowing us to know which code and config were running on an environment.

We don’t need to move all builds along the deployment pipeline — so build 123-abc that was deployed to the test server does not have to go to pre-production automatically as that may be an interim build for QA. If we decide to deploy build 123-abc with the pre-production config to the pre-production server then the build would remain at 123-abc.

We should not rebuild code for different environments; that means the rebuilt code has not been tested on previous environments.
