S03E05 Tiddly pom on a rocket

This week has been about short term and long term flow.

The team sprints are going like rockets this week. Work is (mostly) flowing left to right, sh*t is getting done, and people are asking for more work all the time. All that is great news, and it has shifted the focus onto getting things ready for the new sprint next week.

Backlog refinement

We have now made time for more backlog refinement. There are still some unknowns that may change the priorities before next week’s sprint planning sessions, but on the whole the backlogs are in good shapes.

Speed bumps

As there are new-ish people in the small team we are hitting some impediments where knowledge is in people’s heads. We all have our versions of Brent.

There have been some excellent sessions from the Brents to share their knowledge on topics with others. It has also highlighted the need for an appropriate level of documentation.

What “Kan-be-dan” (say it in a Cockney accent)

Some project workflow is a fairly simple flow focussing on getting work done, and moving it left to right. Some workflows aren’t so simple. Whileit is tempting to tinker with the workflow definitions I think it is better to stop and think about what are we trying to achieve.

Reduce Rework (re)Cycle

Dev work is finished and builds completed in the afternoon GMT so the QA team in India can test when they start the working day at some early hour of the morning GMT-time.

Then at the daily standup we look at the work in progress and discuss any findings from QA. There is a perception that there is a fair amount of rework, but as of yet we have not measured it.

I raised the question on the Slack channel for the South Wales Agile Group and had some interesting comments on whether there should be a “reopened” status, or if there are other ways to track rework. My gut feel is that a story should remain “in progress” while dev and QA work is being done and tools like the Control Chart can be used to identify longer running stories, and then we can look at why that happened.

In order for that to work though, people in the team need to get to grips more with the Lean and Kanban concepts of continual flow. In the shorter term we can get immediate value by introducing a “reopened” status and tracking the amount of tickets passing through that status in a sprint. As we use Jira this can be done by using a query along the lines of

project = WIDGET AND status was Reopened AND sprint in openSprints() ORDER BY priority DESC, key ASC

We can use that in the forthcoming retros to look at the tickets that were reopened and why that was the case.

Looking ahead

As the amount of work being discussed and agreed continues to increase, as well as getting better at backlog refinement and release planning I am looking more at how we can improve productivity (and reduce any rework). This includes looking at how code changes are tested, built and deployed (we have good build pipelines and automatic deployments, but not so great on automated testing or visualising the test and quality metrics). There are some quick wins there already.

Also we need to make more use of “feature flags”. For the new features o changes, or partial features or changes, that are defined we can start work on those, use the flags to turn them on for QA and UAT, and off for production. While those known changes are being made we can then plan the rest of the changes that need to be done to complete the feature set. There is also the benefit of demo’ing the partial changes on a UAT environment to the clients to get earl feedback. Eventually we can get to canary releases, A/B testing and all that good stuff.
