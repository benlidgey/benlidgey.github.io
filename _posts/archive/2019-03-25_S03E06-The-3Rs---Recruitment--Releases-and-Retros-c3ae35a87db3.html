S03E06 The 3Rs — Recruitment, Releases and Retros

I haven’t written week notes for a little while as things have been manic in a good way mostly. Work continues at a pace, dealing with all sorts of different challenges: selecting an office, recruiting developers, planning releases and a whole load of work for new features and defects.

Recruitment

We are recruiting for mobile and web developers, especially those with experience in JavaScript frameworks. We use React and ReactNative a lot, and want to talk to people with React experience, or experience in other JS frameworks like AngularJS or NodeJS.

Our clients range from global, multi-national companies with large systems in multiple languages and locales, to dynamic start-ups pushing the boundaries of innovation.

We rely on people who want to work in a collaborative environment to deliver bespoke solutions using the latest web, mobile and cloud technologies. With an open culture people are freely able to challenge, mentor, help and learn from each other daily. Teams have ownership of the systems they produce, so everyone is encouraged and supported to produce clean, high quality code

More information can be found at https://www.deliveryblueprints.com/who-we-are

Releases

The main point of software development is to provide something useful for the user. And to do this you need to release stuff that works. If you can improve the quality that’s great. If you can improve the speed to match what the user wants then that’s also great.

Improving both quality and speed can be helped by using CI and CD tools. In order to make the tools work though you need to be certain the tools are building and deploying the right things. We have build systems that run regularly several times a day and deploy a build to the test systems.

Recently we had a scenario where we needed to make a bug fix release to the test system and when this was discussed by the team we found there were some commits for a new feature we didn’t want to release on the same branch. We had a review of the branches in use, and created a plan to generate a build for the test system without the partially completed new feature which involved some git jiggery-pokery. This highlighted the need for an agreed branching strategy to be used by the teams, and therefore can be used by the tools.

Similarly the releases need to be planned so it is clear to everyone what is targeted for each release. Everyone can then make sure that whatever they are doing can be linked to the release goals, and challenge tasks when it is not obvious how the link to the release goals.

All of this comes back to people: people being disciplined with the branches they use, people planning releases and dependencies between them (for example this client version needs that server version) and most importantly the communication between all the people on what the goal of the releases are and what the dependencies are between them.

Retrospectives

I ran some retrospectives this week for distributed teams (although the Cardiff team are in the same room). The easiest way I found was to open a blank Word document and share the screen. People can then contribute their thoughts to the 3 sections for “what went well”, “what didn’t go well” and “things to improve”. It has worked well so far, generated some good comments and actions.

Something I noticed was that some people were quiet. This might be because they genuinely didn’t have anything to contribute but I suspect it is because the format does not allow people to reflect and put their tohughts forward, perhaps anonymously. Any ideas on how to manage retrospectives for remote teams gratefully received.

I also tried a different approach with a team that had a sprint with a lot of changing stories, or stories that were not clearly defined. For this one I tried the “Sailboat retrospective” with the idea to not only look at what happened in the previous sprint but to look ahead to the next one and discuss the goal, and what the rocks (risks), anchors (things slowing the team down), and the wind (things helping the team) might look like. To do this I had the picture on a powerpoint slide (again on a shared screen) and frantically wrote notes on the picture in the relevant places. I thought it generated some good conversation, and helped to think about the goals, but that view wasn’t shared by everyone.
